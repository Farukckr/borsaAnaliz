# Plan

## Status

`placeholder | ready | in-progress | blocked | done` plus the date last updated.
The planner sets `ready`. Codex sets `in-progress`, then `done` or `blocked`.

## Task Type

New site / feature / bug fix / refactor / deploy / content update / other.

## User Request

Paste or summarize the exact user request.

## Goal

One sentence describing the user-visible outcome.

## Current State

- Existing files, behavior, and constraints.
- Relevant design, framework, deploy, or data facts.
- Any important risks or unknowns.

## Proposed Changes

- [ ] Concrete implementation steps as a checklist. Codex checks each step off as it completes it, so an interrupted run can resume.
- Mention likely files to create, edit, or leave alone.
- Keep this implementation-ready, not philosophical.
- For large tasks, group steps into numbered phases that each leave the project working. Codex implements one phase per run.

## Acceptance Criteria

- Observable behavior that must be true when done.
- Include responsive, accessibility, deploy, data, or error-state criteria when relevant.

## Verification

- Commands to run.
- Manual checks if commands are unavailable.
- Browser/device checks for UI work.

## Assumptions

- Non-blocking assumptions the implementer may use.

## Open Questions

- Only questions that truly block implementation.

## Risks

- Things that could break or need careful handling.

## Rollback / Recovery

- How to undo or recover if the change goes wrong.

## Out Of Scope

- Tempting but intentionally excluded work.

## Notes For Codex

- Preferred implementation details.
- Existing patterns to preserve.
- Token-saving hints, such as files to ignore.

## Implementation Report

Codex fills this in at the end of a run. The planner reads it when replanning.

- Outcome: done / blocked / partial.
- What changed and which files.
- Verification performed and results.
- Blockers, surprises, or deviations from the plan, with reasons.
- Remaining risk or follow-up.
